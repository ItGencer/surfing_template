# Decisions

Журнал рішень для `Surfing_template`.

## 2026-07-03 - Документація створюється перед реалізацією

**Статус:** accepted

**Рішення:** Спершу створити комплект документації `CODEX.md`, `docs/codex/*`, `docs/design/*`, а вже потім переходити до верстки.

**Причина:** Проєкт має детальний дизайн-промпт і поки майже порожню реалізацію. Документація фіксує вимоги, щоб наступна сесія Codex не втратила контекст.

**Компроміси:** Це не додає видимого UI одразу, але зменшує ризик хаотичної реалізації.

## 2026-07-03 - Дизайн-промпт Surfing Institute є джерелом UI-вимог

**Статус:** accepted

**Рішення:** Вважати доданий дизайн-опис основним джерелом для блокової специфікації сайту.

**Причина:** У ньому вже описані секції, кольори, типографіка, spacing, патерни карток, marquee, FAQ і footer.

**Компроміси:** Частина контенту, фото, шрифтів і footer attribution ще потребує підтвердження.

## 2026-07-03 - Залишити поточний Webpack-стек як базовий

**Статус:** accepted

**Рішення:** Не додавати React, Next.js, Vite або інший фреймворк без окремого рішення. Поточний базис - Webpack + HTML + vanilla JS + Sass.

**Причина:** `package.json` і `webpack.config.js` уже налаштовані для static frontend.

**Компроміси:** Компонентність доведеться підтримувати через HTML/CSS-патерни і невеликий JS, а не через framework components.

## 2026-07-03 - Непідтверджені факти позначати як `to confirm`

**Статус:** accepted

**Рішення:** Не вигадувати реальні URL, контакти, ціни, джерела фото, шрифти, API чи форми. У документації позначати невідоме як `to confirm`.

**Причина:** Користувач попросив не вигадувати факти, яких нема в описі або коді.

**Компроміси:** Документація міститиме відкриті питання, які потрібно закрити перед production-релізом.

## 2026-07-03 - Референси зберігати в `docs/design/references/`

**Статус:** accepted

**Рішення:** Використовувати `docs/design/references/` для скриншотів, фото-референсів, порівнянь desktop/mobile і візуальних нотаток.

**Причина:** Це окрема папка для дизайн-артефактів, щоб не змішувати їх з кодом і markdown-специфікаціями.

**Компроміси:** Папка потребує дисципліни в назвах файлів і періодичного очищення від застарілих референсів.
